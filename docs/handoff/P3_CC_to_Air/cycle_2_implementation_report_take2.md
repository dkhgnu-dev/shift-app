## Kazumaxの仕様判断（着手前に確定した事項）

Dexから「必須確認」とされた2点について、Kazumaxとやりとりし、以下の通り確定しました。

1. **希望休のアーキテクチャ**: 従業員編集モーダルの「希望休（日付テキスト欄）」は残し、初回生成前の事前予約を維持する。加えて、シフトマスのセルで「希望休」を選ぶと、そのセルと従業員のテキスト欄(`requests`)を自動的に同期させる（一本化）。これにより、旧来の「セル側の独立した特殊シフト『希望休』」という二重管理を廃止し、Dexが指摘した衝突の主因を構造的に除去した。
2. **有休の連勤カウント**: ハイブリッドルールを採用。
   - 既存の実働連勤上限（正社員等5連勤／パート等4連勤）のカウントからは有休を除外する（有休を挟めば連勤はリセットされる）。
   - 新たに「7日間の窓の中で真の休み(OFF/公休)が最低1日必要」という法令ハード制約を全雇用形態共通で追加する。有休はこちらのカウントには出勤扱いで含める（例:「4連勤+有休+4連勤」の合計9日間・真の休みなしは禁止）。
   - 応援・店長会・研修・勉強会の4種は、Kazumaxの回答通り引き続き実働連勤カウントに含める（Cycle 2時点から変更なし）。
3. **残存する手動固定と希望休の衝突**（上記1の一本化後も、テキスト欄で希望休登録済みの日にセルで実シフト/有休等を手動固定した場合に発生しうる）: 手動固定（より新しい・明示的な操作）を優先しつつ、必ずwarningとして表示する。

## 必須修正への対応

### 1. [P1] 求解不能時に固定セルが上書きされる → 対応済み

`backend/shift_solver.py`のINFEASIBLEフォールバックを、`fixed_map`を参照するよう書き換えた。固定OFF・固定通常シフト・固定特殊シフトのいずれも、フォールバック時に保持される。デフォルト割当も`allowed_shifts`と`shift_ids`の既知IDのみに限定し、特殊シフトが空欄へ自動割当されないようにした。恒久テスト`test_cycle2_take2.py`の`test_infeasible_fallback_preserves_fixed`で検証（実ブラウザ検証でも、後述の通り実際にこの経路を踏んで動作確認済み）。

### 2. [P1] TimePickerの24:00が00:00になる → 対応済み

`TimePicker`の`setH`/`setM`を、24を`% 24`で丸めず`Math.min(24, ...)`でクランプする実装に変更。24:00到達時は分を0に固定。ブラウザ実機で「23→▲→24→▲(24で停止)」「24をタップ→24:00維持」を確認済み。

### 3. [P1] fixed_assignmentsと強制希望休の優先順位 → 対応済み（Kazumax判断済み）

上記「Kazumaxの仕様判断」の通り、手動固定を優先しつつ`shift_solver.py`が明示的なwarningを生成するよう実装。ブラウザ実機で実際に衝突を発生させ、`「T.S.さん: 6/20は希望休が登録されていますが、手動固定「④」を優先しました。」`という警告が表示されることを確認済み。

### 4. [P2] 不正な固定割当を黙って無視しない → 対応済み

`shift_solver.py`に検証ロジックを追加。未知employee、未知shift、期間外day、同一セルの重複を検出し、それぞれ`warnings`に明示的なメッセージを追加した上でスキップ（重複は後勝ち）。恒久テストの`test_invalid_fixed_assignments`で4パターンとも検証。

### 5. [P2] 特殊シフト時間を0〜24hへ実処理でも制限する → 対応済み

`applySpecialHours`に保存時バリデーションを追加。`Number.isFinite`かつ0〜24の範囲外の場合は保存せず、`alert`でエラーを表示する。ブラウザ実機で30hを入力→保存拒否・モーダル維持を確認済み。

### 6. [P2] 表示上の初期時刻とstateの不一致 → 対応済み

`openModal()`が`customStartTime`/`customEndTime`を`''`にリセットしていた箇所を`'09:00'`/`'18:00'`に修正。ブラウザ実機で、ピッカーを一切操作せず「この時間を追加」を押しても正しく`09:00～18:00`（実際にはテスト時は片側24:00に調整した`09:00～24:00`）が追加されることを確認済み。

### 7. [P2] Cycle 2の恒久回帰テストを保存する → 対応済み

`backend/test_cycle2_take2.py`を新規作成し、以下6シナリオ・16アサーションを再実行可能なテストとして保存した（Dex指定の7シナリオのうちTimePicker/特殊時間の境界値はフロントエンドのUIロジックのため、本テストの対象外としブラウザ実機検証で担保）。

3. INFEASIBLEフォールバック時も固定シフトが保持される（意図的に7日間休みなしの矛盾を発生させて検証）
4. fixedと強制希望休の衝突時、手動固定が優先されwarningが記録される
5. 未知employee/未知shift/期間外day/重複fixedがクラッシュせず、それぞれwarningとして記録される
6. 特殊シフト固定時、店舗シフトへ自動割当されず店舗集計から除外される

## まだ不安な点・Dexに特に見てほしい点

1. **7日間休み必須のハード制約の対象範囲**: 「全雇用形態共通」で一律適用しましたが、Kazumaxの説明は一般的な例示だったため、雇用形態によって除外すべきケース（例: 時間限定社員等）がないか、再確認をお願いしたいです。
2. **希望休の一本化に伴うUX**: セルで「希望休」を選ぶとテキスト欄と自動同期されますが、テキスト欄側を直接編集した場合は`saveEmployee`が生成結果を破棄する既存挙動のため、その後の再生成が必要です。この非対称性がわかりにくくないか、UI観点でのご意見をいただきたいです。
3. **INFEASIBLEフォールバックへの遷移頻度**: 今回の実機検証で、たった1セルの手動変更でもPhase1がINFEASIBLEになるケースに遭遇しました（空欄自動作成でほぼ全セルが固定されている状態のため、1日の出勤人数±1名制約などと衝突しやすいと考えられます）。恒久バグではありませんが、実運用でフォールバック頻度が高くなる可能性がある点は留意事項として共有します。
