# Shift-Ag プロジェクト現状棚卸し (CURRENT_STATUS)

## 1. 何を作りたかったのか（目的）
* ドラッグストア等の小売店舗を対象とした、**制約満たし型（条件を考慮した）の自動シフト作成プラットフォーム**。
* 登録販売者の配置ルール、希望休、契約日数、曜日ごとの厚み付けといった複雑な条件をAI（数理最適化ソルバー）が計算し、ボタン一つで最適な1ヶ月のシフトを自動生成する。
* スマホやPCから直感的に操作できるモダンで美しいUI（ガラスモーフィズム等）を提供する。

## 2. 今何ができているのか（実装済み機能）
### バックエンド (FastAPI + OR-Tools)
* Google OR-Tools (CP-SATソルバー) を用いたシフト最適化ロジック（`shift_solver.py`）。
* 制約のモデリング（契約日数の厳守、希望休の反映、登録販売者の配置、固定シフトの割り当て）。
* ポートフォリオ検索（並列処理）による複雑な制約のタイムアウト回避。
* GitHub -> Renderへの自動デプロイ環境。

### フロントエンド (React + Vite)
* 全体シフト表、従業員管理、ルール設定の3つの主要タブ構成。
* 生成結果や従業員データの **LocalStorage (ブラウザ内) 保存** 機能。
* ドラッグ＆ドロップによる直感的な従業員の並び替え（スマホ向けは上下ボタンでの補助）。
* 表示月/年の動的切り替え（カレンダーの曜日や日数が自動連動）。
* 「①」等のIDではなく「8:15～12:15」といった具体的な時間帯でのシフト表示。
* スマホ向けのレスポンシブデザイン（ハンバーガーメニュー搭載）。
* GitHub -> Vercelへのデプロイ構成。

## 3. 何ができていないのか・どこで詰まっているのか
1. **制約が厳しすぎる場合のエラーハンドリング（詰まりポイント）**
   * 希望休が被りすぎている、または登録販売者が不足しているなどの理由で「理論上作成不可能なシフト」になった場合、全体がエラー（作成失敗）になってしまう。
   * 「一部だけ空欄にしてとりあえず作る」といった**部分的なシフト生成（ソフト制約化）**が未実装。
2. **データの永続化と共有（未実装）**
   * 現在のデータ保存はブラウザのLocalStorageに依存しているため、店長がスマホで作ったシフトを店舗のPCで見る、といった「端末間のデータ共有」ができない。
   * PostgreSQLなどのデータベース連携が未実装。
3. **UI上の微調整**
   * 生成されたシフトをCSVやPDF等で出力（ダウンロード）する機能がない。

## 4. 今の作り方が合っているのか（アーキテクチャの評価）
* **数理最適化 (OR-Tools) の採用：** ◎ 大正解。シフト作成のような複雑な制約解決問題において、機械学習ではなくCP-SATソルバーを用いたのは非常に適切であり、計算速度と正確性が担保されている。
* **React + Vite のフロントエンド：** ◎ コンポーネント指向でUIの拡張がしやすく、現在の構成は最適。
* **データ管理（LocalStorage）：** △ MVP（初期プロダクト）としては開発スピードが上がり良かったが、実運用フェーズに入るならDB化が必要不可欠。

## 5. 次にどこだけ直すのがよいか（Next Actions）
優先度の高い順に以下の改修を推奨する。

1. **DB連携の導入 (Supabase または SQLite等)**
   * LocalStorageからDBへ移行し、複数端末でのログインとデータ共有を可能にする。
2. **ソルバーのソフト制約化（エラー回避策）**
   * 制約違反を「エラーで失敗」させるのではなく、「ペナルティスコアを与えつつ、警告マークを出してシフト自体は返す」ように `shift_solver.py` を改良する。
3. **シフトのエクスポート機能**
   * 確定したシフトを印刷したり従業員に配れるよう、CSVまたは画像/PDFでの出力ボタンをフロントエンドに追加する。
