# P4 レビュー依頼: Cycle 1（CC担当分）+ main統合内容

## Dexへ

Dex制限中はエアクルーが代役でしたが、復帰されたとのことで、Cycle 1でCCが実装した内容と、その後mainで発生した友達側AIエージェント「アグ」との並行開発・統合について、あらためてレビューをお願いします。

## 何をしたか

1. **Cycle 1実装**（`docs/handoff/P2_AirCrew_to_CC/cycle_1_mobile_and_flexible_shifts_instructions.md`に基づく）
   - `backend/shift_solver.py`: `shift_types`の`start_time`/`end_time`から時間帯ブロック(B1〜B10)を動的判定する仕組み、前月16日〜当月15日締めの期間・曜日判定
   - `frontend/src/App.jsx`: スマホ用ハンバーガードロワー、`shiftMaster`によるカスタムシフトパターン管理・自由時間入力、16日〜15日締めの日付/曜日表示
   - `frontend/src/index.css`: モーダルz-index調整
   - 詳細は `docs/handoff/P3_CC_to_Air/cycle_1_implementation_report.md` 参照

2. **main統合作業**（重要・危険領域）
   - Cycle 1実装中に、友達側「アグ」がリアルタイムかつforce-push込みで`main`に並行して別機能（辞書式最適化・スラック自動診断エンジン、v4.8〜v4.10）をpushしていることが判明。`shift_solver.py`/`App.jsx`でファイルレベルの衝突が発生。
   - Air判断で`air-cc-dev`ブランチを新設しCC作業を隔離。その後Kazumaxさんの調整でアグ側が一時的に作業を停止した間に、`git merge --allow-unrelated-histories`で両者を手動統合し、`main`へfast-forward push。
   - 統合時に見つけたバグ修正：アグ側コードに残っていた `emp.days`（存在しないフィールド）参照を `emp.contract_days` に修正。
   - 統合後もアグ側は継続的に`main`へpushしており（v4.8→v4.9→v4.10）、その都度追従マージ・ビルド確認・pushを実施済み。現時点で`air-cc-dev`と`main`は同一コミットを指しています。

3. **デフォルト従業員データの実データ化**
   - 友達の実店舗の従業員構成（写真から取得：雇用区分・登録販売者資格・契約日数・シフト時間帯）を`INITIAL_DATA`に反映。
   - 氏名は個人情報保護のため実名ではなく、写真の氏名から推定したイニシャル表記（例: `K.D.`、重複時は`(1)(2)`で区別）を使用。読みが確定しづらい漢字は推定のため、イニシャルの正確性は保証できません。

## まだ不安な点・Dexに特に見てほしい点

1. **シフト自動生成ロジックへの継続的な変更**（危険領域）：`shift_solver.py`はアグ側が単独でv4.8〜v4.10まで頻繁に更新しており、CC側は都度追従マージしているだけで、アグ側変更の設計意図・妥当性そのものは検証していません。特に連勤・連休制限まわりのハード制約が積み重なっており、制約同士の矛盾（解なし）が発生しないか要確認です。
2. **「曜日ごとの目標人数が平均±1名の枠に縛られて現場の実態と合わない」という設計上の課題**（友達からのフィードバック、未実装）：月内平均から算出した1つの目標値に対して曜日順位で±1しか動かせないため、「均等すぎる」「逆に偏る」という symptom が出ています。原因分析はしましたが実装はまだしていません。曜日ごとに実数で目標人数を直接指定できる仕組みへの変更を提案していますが、Dexの意見も伺いたいです。
3. **未修正の既知バグ（実装未着手）**：
   - a) デフォルト従業員データを更新しても、一度でも`localStorage`に従業員データが保存されたブラウザ/端末では古いデータが表示され続ける（新デフォルトが反映されない）。特にPCとスマホでlocalStorageが別々なため、デバイスごとに個別のクリアが必要になっている。恒久対策として「初期データにリセット」ボタンやデータバージョニングの導入を検討中。
   - b) 「ルール設定」タブの「シフトパターン管理」一覧で、自由時間シフト（IDと表示文字列が同一の長い文字列）が `id : timeStr` の形で重複表示され、スマホ幅で枠からはみ出す。従業員一覧テーブルでは同種の重複表示バグを既に修正済みだが、この一覧には未適用。
4. **希望休の意味変更**：`requests`（希望休入力欄の数字）が「月内の日付」から「対象期間(前月16日始まり)内の通算番号」に変わっている点（Cycle 1 P3報告書に記載済み、再掲）。
