# WORKFLOW_RULES - 友達シフト連携ルール

## アイ単体の作業フロー

```text
友達/Kazumax -> アイ -> GitHubへpush -> 必要時Kazumax側Dexレビュー -> アイが修正
```

1. アイは作業前に `git pull --ff-only` で最新版を取得する。
2. `AGENTS.md` と `CURRENT_STATUS.md` を読む。
3. 作業内容が危険でないか判断する。
4. 実装または棚卸を行う。
5. 結果を `docs/handoff/` に保存する。
6. 検証内容を書いてcommit/pushする。

## Dexレビューが必要なとき

以下の場合は、Kazumax側Dexにレビューを依頼してください。

- シフト自動生成ロジックに触った
- DB、保存形式、既存データに触った
- 変更範囲が大きい
- 何度直しても思い通りに動かない
- アイが不安を感じる
- Kazumaxまたは友達が「Dexにも見てもらって」と言った

## Dexへ渡す報告に含めること

Dexレビューを依頼する場合、`docs/handoff/` に以下を保存してください。

- 何をしたか
- どのファイルを変更したか
- どう確認したか
- まだ不安な点
- Dexに特に見てほしい点

## Kazumax/Dex側の作業フロー

Kazumax側Dexがこのプロジェクトを見る場合は、必ず作業前に最新版をpullしてください。
レビュー結果や修正指示は `docs/handoff/` に保存し、必要ならcommit/pushしてください。

## 人間確認を減らすルール

人間に「全部確認して」と丸投げしないでください。
確認が必要な場合は、見る場所を1〜5個に絞ってください。

- 確認不要: 文書整理、軽微な内部整理
- 軽い確認: UIや通常導線の小さな変更
- 必須確認: シフト自動生成ロジック、保存、削除、本番利用、個人情報

## branch共同作業ルール

複数人・複数AIで同時に作業する場合は、mainへ直接作業せず、作業用branchを使ってください。

考え方:

- `main`: 完成版・安定版の置き場
- `branch`: 作業中の机

作業開始:

1. `main` に移動する。
2. `git pull --ff-only` で最新版を取る。
3. 作業用branchを作る。
4. 触る予定ファイルと、衝突しそうな場所を確認してから作業する。

作業完了:

1. 作業branchでcommitする。
2. 作業branchをpushする。
3. 必要ならDexレビューに回す。
4. OKになったら、mainをpullして最新版にする。
5. 作業branchをmainへmergeする。
6. mainをpushする。

mergeのタイミング:

- 1サイクルが完了したとき。
- レビューOKになったとき。
- 人に見せられる小さい完成単位になったとき。
- 文書やhandoff追加は早めにmergeしてよい。
- シフト自動生成ロジック、保存形式、DB、大きなUI変更はレビュー後にmergeする。

衝突を減らすルール:

- 同じファイル・同じ機能を同時に触らない。
- 特に `frontend/src/App.jsx`、`backend/main.py`、`backend/models.py`、`docs/handoff/CURRENT_STATUS.md`、`README.md`、`package.json`、`requirements.txt` は衝突しやすい。
- `CURRENT_STATUS.md` は、できればmainへmergeする担当が最後に更新する。

conflictが出た場合:

- 勝手に無理やり解決しない。
- 衝突ファイルを報告する。
- main側の内容とbranch側の内容を説明する。
- 採用案、統合案、おすすめ理由を出す。
- 重要な衝突は、人間またはDexの判断を待つ。
- 解決後は `<<<<<<<`, `=======`, `>>>>>>>` が残っていないか確認する。
