# 友達シフト AI運用テキストブック

このテキストは、友達シフト作成ツールを「アグ」と一緒に進めるための人間向け説明書です。
難しいGitやAI運用を全部覚える必要はありません。
まずは「何を頼めばいいか」「どこに結果が残るか」「困ったらどう止めるか」だけ分かれば大丈夫です。

## 1. 何のためのマニュアルか

このプロジェクトでは、アグに作業を頼む前に、いくつかのルールを読んでもらいます。
目的は、アグがいきなりコードを書き始めたり、今の状況を勘違いしたり、危ない変更を勝手に進めたりしないようにすることです。

マニュアルは「アグへの説明書」です。
人間が毎回長く説明しなくても、アグが同じルールで動けるようにするために置いてあります。

## 2. 最初に読むファイル

アグには、作業前に以下を読ませます。

- `AGENTS.md`
- `docs/handoff/CURRENT_STATUS.md`
- `docs/PROJECT_RULES.md`
- 必要に応じて `docs/handoff/WORKFLOW_RULES.md`

それぞれの役割はこうです。

- `AGENTS.md`: アグが最初に読む入口。作業の基本ルール。
- `CURRENT_STATUS.md`: 今どこまで進んでいるか。
- `PROJECT_RULES.md`: シフト作成ツール専用の注意点。
- `WORKFLOW_RULES.md`: GitHub連携やDexレビューの詳しいルール。

## 3. 友達が覚えること

友達が全部を覚える必要はありません。
最低限これだけで大丈夫です。

1. 作業前にアグへ「マニュアルを読んで」と言う。
2. いきなり実装させず、まず現状を整理させる。
3. 長い報告は `docs/handoff/` に保存させる。
4. 改善案は `docs/proposals/` に保存させる。
5. 作業が終わったらGitHubへ反映させる。
6. 困ったら無理に進めず止まって報告させる。

## 4. GitHubの考え方

GitHubは、みんなで見られる共通の棚です。

- `pull`: GitHubから最新版を取ってくること。
- `push`: 自分の変更をGitHubへ戻すこと。
- `commit`: 変更をひとまとまりの記録にすること。

イメージはこれです。

```text
友達のPC
  ↓ push
GitHub
  ↓ pull
Kazumax側Dex
```

アグには、作業前に最新版を取り、作業後に結果を戻してもらいます。
友達が毎回Gitの細かい操作を覚える必要はありません。

## 5. アグに最初に送る文章

最初はこの文章をアグに貼ってください。

```text
アグへ：

このプロジェクトはGitHubで共有しています。
まず最新版を取り込んでから作業してください。

最初に読むファイルは以下です。

- AGENTS.md
- docs/handoff/CURRENT_STATUS.md
- docs/PROJECT_RULES.md
- 必要なら docs/handoff/WORKFLOW_RULES.md

読んだら「読了報告」「今どこまで進んでいるか」「今回何をするか」を短く教えてください。

今回は実装ではなく、途中棚卸をお願いします。
今のシフト作成ツールについて、何を作りたかったのか、今何ができているのか、何ができていないのか、どこで詰まっているのか、今の作り方が合っているのか、次にどこだけ直すのがよいかを整理してください。

整理した内容は docs/handoff/ に保存してください。
最後にGitHubへ反映できるならcommit/pushしてください。
できなければ、どこで止まったか教えてください。
```

## 6. 途中棚卸とは

途中棚卸は、作りかけのプロジェクトを一度整理する作業です。
うまくいかないときに、いきなり作り直すのではなく、まず原因を見つけます。

確認することはこれです。

- 最初に作りたかったものは何か。
- 今できていることは何か。
- 今できていないことは何か。
- どこで詰まっているか。
- 今の作り方が目的に合っているか。
- 残す部分と捨てる部分はどこか。
- 次の1回で直すなら、どこだけ直すのがよいか。

シフト作成ツールは、希望休、必要人数、役割、連勤、公平さなど条件が多いです。
だから、最初に棚卸をすると迷子になりにくくなります。

## 7. Dexに見てもらうタイミング

普段はアグ単体で進めて大丈夫です。
ただし、以下のときはKazumax側Dexにレビューしてもらうと安全です。

- シフト自動生成ロジックを大きく変えた。
- 保存形式やDBに触った。
- 変更範囲が大きい。
- 何度直しても思い通りに動かない。
- アグ自身が不安そう。
- 友達やKazumaxが「Dexにも見てもらおう」と思った。

Dexに見てもらうときは、アグに作業報告を `docs/handoff/` に保存してpushしてもらいます。
その後、Kazumax側DexがGitHubから最新版を取ってレビューします。

## 8. やってはいけないこと

以下は危ないので、アグに勝手に進めさせないでください。

- `.env`、パスワード、APIキーをGitHubへ入れる。
- 本物の個人情報やスタッフ情報をそのままGitHubへ入れる。
- 本物データを勝手に削除する。
- conflictが出たのに無理やり直す。
- 大きな作り直しを相談なしで始める。

危ないと思ったら、アグには止まって報告してもらってください。

## 9. 普段の頼み方

棚卸が終わった後は、このように頼めます。

```text
アグへ：

まず最新版をpullして、AGENTS.md と CURRENT_STATUS.md を読んでください。
今回の作業内容を短く確認してから進めてください。
完了したら、変更内容と確認結果を docs/handoff/ に保存し、commit/pushしてください。
```

Dexレビューが必要なときは、こう頼めます。

```text
アグへ：

今回の作業はDexレビューに回したいので、変更内容、確認結果、不安な点、Dexに見てほしい点を docs/handoff/ に保存してからpushしてください。
```

## 10. まとめ

友達がやることはシンプルです。

1. アグに最初の読むファイルを伝える。
2. まず棚卸をさせる。
3. 結果を `docs/handoff/` に残させる。
4. GitHubへpushさせる。
5. 必要なときだけKazumax側Dexにレビューしてもらう。

この流れにすると、友達のPC、GitHub、Kazumax側Dexが同じ情報を見ながら進められます。

## 11. branch作業の考え方

共同作業が増えたら、mainへ直接作業せず、branchを使うと楽になります。

イメージはこうです。

- `main`: みんなが見る完成版の棚
- `branch`: それぞれの作業机

アグが作業するときは、たとえば `agu/cycle1-inventory` のようなbranchで作業します。
Kazumax側Dexがレビューするときは、必要に応じて `dex/review-cycle1` のようなbranchを使います。

作業の流れはこうです。

```text
mainを最新版にする
↓
作業用branchを作る
↓
作業する
↓
commit/pushする
↓
レビューOKならmainへmergeする
↓
mainをpushする
```

大事なのは、merge前にもmainを最新版にすることです。
自分が作業している間に、相手がmainへ更新を入れているかもしれないからです。

```text
merge前:
mainへ移動
↓
git pull --ff-only
↓
作業branchをmerge
↓
git push
```

mergeは「時間」だけで決めるより、「小さく完成したタイミング」で行うのがおすすめです。

- 棚卸レポートが完成した
- 小さいバグを直した
- 画面1つを直した
- レビューOKになった
- 文書やhandoffを追加した

逆に、まだ壊れている、途中で動かない、保存形式や自動生成ロジックが未確認、という場合はmainへ入れない方が安全です。

## 12. conflictが出たとき

branchを分けても、同じファイルの同じ場所を別々に変えるとconflictが出ます。
conflictは「どっちの変更を採用するか相談して」というGitからの合図です。

たとえば同じ文を別々に変えた場合、ファイルにはこう出ます。

```text
<<<<<<< HEAD
main側の内容
=======
branch側の内容
>>>>>>> agu/cycle1
```

この場合、アグは勝手に無理やり直さず、こう報告してください。

```text
conflictが出ました。
衝突ファイル:
- frontend/src/App.jsx

main側の内容:
...

branch側の内容:
...

案:
- main側を採用
- branch側を採用
- 両方を統合

おすすめ:
...
```

特に注意するファイルはこれです。

- `frontend/src/App.jsx`
- `backend/main.py`
- `backend/models.py`
- `docs/handoff/CURRENT_STATUS.md`
- `README.md`
- `package.json`
- `requirements.txt`

作業前に「今回触る予定ファイル」を出しておくと、衝突はかなり減らせます。
